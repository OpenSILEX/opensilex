# Migrate to 1.5.0

## Deprecated API
 
Notice that some API paths are deprecated and will be removed. Ensure your client applications do not use these paths anymore. The deprecated paths are:
- everything in `/security/users`
- POST `/core/data/by_targets`
- GET `/core/data/export`
- GET `/core/data`

## Modifications to CSV import formats

The Scientific Objects CSV import format has changed slightly in this new version. vocabulary:isHosted and hasGeometry are no longer
accepted columns. Instead, you can use a set of columns that you would normally find in a MoveEvent import, these columns are:
- start_date_of_Location, the optional moment where the Scientific Object began to have this location.
- end_date_of_Location, the moment where the Scientific Object no longer has this Location. Just like a Move Event import, this one is required, if an End is filled but not a Start, then the Move is considered to be instantaneous.
- from, the optional Facility where the Scientific Object is being moved away from.
- to, the optional Facility where the Scientific Object is being moved too. This column replaces the old isHosted.
- coordinates, optional location in wkt format. This one replaces the old hasGeometry.
- x, y, z, textualPosition (4 columns) to provide extra information in textual format about the Move, just like a Move Event import

## Reset ontologies

Modifications have been made to the core ontologies in this version. You will need to update the ontologies on your triple store:

[! WARNING] > you should do this BEFORE running the migration command

```
java -jar opensilex.jar --CONFIG_FILE=<config_file> sparql reset-ontologies
```

## JVM manipulation for RDF4J servers

If you are using RDF4J as a triple store, you will need to increase the JVM stack size to avoid `StackOverflowError` when performing SPARQL queries.

You are using the provided docker compose in `opensilex-dev-tools/src/main/resources/docker/docker-compose.yml` ? We have updated the docker-compose file. you just have to refresh the RDF4J image by running:
```bash
cd <path_to_opensilex>/opensilex-dev-tools/src/main/resources/docker
docker compose up -d
```

It should say that MongoDb is running and RDF4J is created.

If you are using your own RDF4J server, you will need to manually increase the JVM stack size by running this command

```bash
java -Xss8m
```

no particular output confirmation is expected.

## Migrations

- To perform all the migrations required to upgrade to 1.5.0, you have to run this single command

```shell
# Connect to a console in OpenSILEX server
# From the directory which contains your OpenSILEX executable .jar file
java -jar opensilex.jar --CONFIG_FILE=<config_file> system run-update org.opensilex.migration.one_point_five_ALL.MigrateToOnePointFive
```

Doing this will apply the following Migrations.

### Migration of existing scientific objects which are created from Experiment with a relation: 'participatesIn'

- The command will add a relation between all Scientific Objects and Experiment when we create these from Experiment. If it is not done then there will be inconsistency in terms of SO data.

### Migration of Facilities location models:

- The command will make sure the Facilities are using all the new Location systems.

See the migration `org.opensilex.migration.one_point_five_ALL.UpdateFacilitiesWithLocationObservationCollectionModel`
from [migration_command.md](../how-to/migration_command.md#orgopensilexmigrationone_point_five_allupdatefacilitieswithlocationobservationcollectionmodel) for additional notes

### Migration of Moves and ScientificObjects location models

- The command will make sure that ScientificObjects and Moves now use the new Location system.

### Migration of new links between Facilities, Variables and Devices

- The command will create shortcut connexions in between Facilities, Variables and Devices.

See the migration `org.opensilex.migration.one_point_five_ALL.FacilitiesLinkToVariablesAndDevicesMigration`
from [migration_command.md](../how-to/migration_command.md) for additional notes

### Migration of types parameter

- This migration change the uri of type parameters in order to separate them from their associated type (when manually created a type with ontologie API or interface).
